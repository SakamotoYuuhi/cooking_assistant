# AI料理アシスタント 開発ロードマップ

## 概要

「毎日使えるAI料理アシスタント」を題材に、実務直結のAI開発スキルを段階的に習得するロードマップ。

---

## フェーズ1：AI料理アシスタント（最重要・最初に着手）

### 目標
毎日実際に使えるAI料理アシスタントを構築する。

### 主要機能

| 機能 | 概要 |
|------|------|
| 食材入力 | 冷蔵庫にある食材をテキストで入力 |
| 献立提案 | 入力食材をもとに献立を提案 |
| レシピ生成 | 具体的な調理手順を生成 |
| 栄養バランス考慮 | PFCバランスなどを加味した提案 |
| 調理時間制約 | 「30分以内」などの時間条件に対応 |
| 会話履歴保持 | 前の会話を踏まえた連続的なやりとり |

### 習得できる技術

- **Bedrock** - AWS上でのLLM呼び出し
- **Prompt設計** - 指示の精度を上げるプロンプトエンジニアリング
- **会話メモリ** - セッション管理・履歴の保持方法
- **API設計** - バックエンドAPIの構造設計
- **UI** - フロントエンドとの接続

---

## フェーズ2：RAG導入

### 目標
自分だけの知識ベースを構築し、検索精度の高い回答を実現する。

### 活用できるデータソース例

- お気に入りレシピ（個人メモ）
- クックパッドの保存レシピ
- Notionのメモ
- Markdownで書いたレシピ集

### ポイント

> **これは実質「社内ナレッジ検索」と同じ構造。**

RAGを自分のレシピで実装することで、企業の社内FAQ・ドキュメント検索に直接応用できる設計思想が身につく。

### 習得できる技術

- Embeddingによるベクトル化
- ベクトルDBの設計・運用（例：OpenSearch, Pinecone, pgvector）
- 検索精度の評価・改善

---

## フェーズ3：エージェント化

### 目標
単一のチャットボットから、複数の専門エージェントが協調するシステムへ進化させる。

### エージェント分割例

| エージェント | 役割 |
|------------|------|
| 在庫確認エージェント | 食材の残量・期限を管理 |
| 栄養分析エージェント | 食事の栄養素を計算・評価 |
| 献立プランナー | 週単位の献立を自動生成 |
| 買い物リスト生成 | 必要な食材をリスト化 |
| 調理順序最適化 | 複数料理を効率よく同時進行 |

### 習得できる技術

- LangGraph的なマルチエージェント設計
- エージェント間の状態管理・ルーティング
- ツール呼び出し（Tool Use）パターン

---

## フェーズ4：業務転用

### 目標
料理アシスタントで培った設計を、実際の業務課題へ応用する。

### 転用先の例

| 業務用途 | 料理での対応機能 |
|---------|----------------|
| 社内FAQ対応 | レシピQ&A |
| チケット整理 | 献立プランニング |
| 手順検索 | 調理手順生成 |
| 障害対応支援 | 調理トラブルシューティング |

### 転用がスムーズな理由

料理アシスタントで実装した構造（入力→検索→生成→応答）は、業務AIと完全に同じアーキテクチャ。題材が変わるだけで、設計思想はそのまま使える。

---

## なぜ「料理」が最強の開発テーマなのか

### 継続改善ができる

AIエージェント開発で差がつくのは**「実際に運用しながら改善した経験」**。

毎日料理をしているなら、毎日フィードバックを得て改善できる。これは他のテーマでは得られにくい強さ。

### 面接・社内提案で語れるエピソードになる

「AIを知っている人」より「実際に運用して改善した人」の方が企業に評価される。

以下のような具体的な話ができるようになる：

- どのように精度を評価したか
- Hallucinationをどう抑えたか
- コストをどう削減したか
- プロンプトをどう改善したか
- RAGの検索精度をどう向上させたか

---

## 開発フェーズ全体像

```
フェーズ1：基礎実装
  └─ Bedrock × 会話AI × UI

      ↓

フェーズ2：知識強化
  └─ RAG × ベクトルDB × 独自レシピ

      ↓

フェーズ3：エージェント化
  └─ マルチエージェント × LangGraph設計

      ↓

フェーズ4：業務転用
  └─ 社内FAQ / ナレッジ検索 / 障害対応支援
```
